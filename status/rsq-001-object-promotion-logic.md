# RSQ-001 — Object Promotion Logic

## Repository Structure Question

**Question:** The repository appears to apply different promotion logics to different object classes. These logics were not specified in advance and emerged through case-by-case decisions. Can the implicit criteria be extracted, specified, and applied coherently to future classification decisions?

**Date opened:** 2026-06-24
**Triggered by:** Post-Checkpoint #007 analysis

---

## Observation

The repository has been promoting and deferring objects using at least three distinct implicit logics:

| Object | How it advanced |
|---|---|
| Adaptation | Survived discriminating tests and pre-registered wound conditions |
| Watch B | Recurrence across domains + accumulation + naming |
| Attribution Precision Gap | Deferred pending boundary specification from adjacent object |
| Four-cell map | Explicit predictive upgrade criterion before promotion considered |

These are not the same decision rule. The repository discovered them case-by-case without naming the distinction.

**Hazard:** Retrospective reconstruction may overstate the coherence of prior decisions. Extracted criteria should be treated as candidate specifications that fit observed behavior — not discovered governing principles that were already operating.

---

## Resolution Path

1. Review prior promotion and non-promotion decisions.
2. Extract candidate criteria that best explain those decisions.
3. **Record candidate criteria before application.** *(Pre-registration step — required to preserve falsifiability.)*
4. Apply criteria to a live object-classification decision.
5. Evaluate whether the specification produced useful operational output.

**Designated test case:** Attribution Precision Gap promotion decision.

---

## Pre-Registered Candidate Specification

*Recorded 2026-06-24 before application to designated test case.*

### Class 1 — Explanatory Objects
*(Selector variants, Adaptation, Technical-only model, Discovery infrastructure)*

Advance by surviving pre-registered wound conditions; generating testable ordering predictions; being pressured by discriminating tests designed to fail them.

Promotion logic: confidence accumulates through test survival, not instance accumulation. One instance surviving a genuine wound condition is stronger than three instances facing no discriminating test.

Demotion: failure of a pre-registered prediction under conditions that preclude ad hoc defense.

### Class 2 — Epistemic Structure Objects
*(Watch B, Attribution Precision Gap, Implementation-documentation limitation, Counterfactual unavailability)*

Advance by recurrence across domains and structural consistency across instances.

Promotion requires all three:
1. Sufficient instance count across distinct domains
2. Structural consistency — the bottleneck takes the same form across instances
3. Boundary specification — explicit differentiation from the nearest adjacent epistemic object

*Note on Watch B:* Watch B was promoted before the three-criterion framework was articulated. It met criteria 1 and 2 at promotion. Criterion 3 (boundary specification from Attribution Precision Gap) was only specified retroactively at Checkpoint #007. This is the expected behavior — categories are often defined by contrast, and contrast requires a second object to distinguish from.

### Class 3 — Observation Maps
*(Four-cell discovery timing map)*

Advance by occupancy — cases filling cells with observed implementations.

Upgrade criterion (named in Checkpoint #007): cell classification must be made before the outcome is known, and the resulting prediction must hold.

Not promotable to explanatory model until predictive relevance is demonstrated prospectively.

## Application to Designated Test Case

**Test case:** Attribution Precision Gap promotion decision.

| Criterion | Status |
|---|---|
| Criterion 1 — Instance count across domains | Satisfied: 3 domains (child survival RC-023, community health RC-028, agricultural development RC-034) |
| Criterion 2 — Structural consistency | Satisfied: outcome accepted, cause accepted, contribution share disputed across all three |
| Criterion 3 — Core boundary specification | Satisfied: Watch B (cause uncertain) vs. Attribution Precision Gap (contribution share uncertain) distinction specified with different evidential requirements |
| Criterion 3 — Edge boundary specification | Incomplete: two unresolved boundary conditions identified (see below) |

**Boundary conditions surfaced by application:**

**Boundary Condition A — Watch B ↔ Attribution Precision Gap (Intensity Threshold)**
Question: At what point does a very small accepted contribution cease to be a decomposition question and become a causal-role question? Current state: Unspecified.

**Boundary Condition B — Attribution Precision Gap ↔ Counterfactual Unavailability**
Question: How should attribution precision be distinguished from situations where the necessary comparison world cannot exist or be observed? Attribution Precision Gap assumes an observable outcome and accepted contributors; counterfactual unavailability concerns whether the comparison world exists at all. Relationship: Unspecified.

These boundary conditions were not present in the extracted candidate criteria. They were surfaced by application. This is evidence against overfitting — a specification that closes completely on first application to a novel case was probably derived with that case in view.

## Result

**Outcome: Useful + Incomplete**

| Test matrix outcome | Result |
|---|---|
| Wrong — specification fails to explain prior or current decisions | No |
| Redundant — specification makes existing logic visible but adds no operational capability | No |
| Incomplete — specification explains part of the decision space but requires extension | Yes |
| Useful — specification changes or materially clarifies future classification decisions | Yes |

The specification produced a materially clearer classification decision than would have been made without it. The decision is now legible: which criteria are satisfied, which are partially satisfied, and what specifically remains open. Two previously unrecognized boundary conditions were surfaced.

**Current status:** Useful first-pass specification with identified boundary incompleteness. Existing repository status labels (Supported, Unresolved, etc.) do not cleanly describe this state — those were built for claims and audits, not repository-structure specifications. This vocabulary gap is noted and not resolved here.

The specification remains a working document, not a governing principle.
